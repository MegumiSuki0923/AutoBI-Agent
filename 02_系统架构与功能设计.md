# 系统架构与功能设计

## 总体架构

AutoBI Agent 采用轻量级前后端分离架构：

- 前端：Streamlit，用于输入自然语言问题、展示 SQL、表格、图表和分析结论。
- 后端：FastAPI，提供统一问数接口和模块编排。
- 数据库：DuckDB 或 SQLite，存储模拟汽车营销销售数据。
- RAG：检索数据字典、指标口径、表关系和示例 SQL。
- LLM：完成意图识别、Text-to-SQL、结果分析和图表推荐。
- SQL Executor：执行安全 SQL 并返回结构化结果。
- SQL Guard：校验 SQL 是否安全、只读、可执行。
- History Store：保存问数历史、SQL、结果摘要和图表配置。

## 数据流

1. 用户在 Streamlit 前端输入自然语言问题。
2. 前端调用 FastAPI 的 `/api/ask` 接口。
3. 后端用 Pydantic 校验请求参数。
4. Agent Orchestrator 接收问题，调用意图识别模块判断问题类型。
5. RAG 模块从数据字典、指标口径、示例 SQL 中检索相关上下文。
6. Text-to-SQL 模块将用户问题、表结构、指标定义和约束规则输入 LLM，生成 SQL。
7. SQL Guard 使用 sqlglot 校验 SQL，只允许安全的 SELECT 查询。
8. SQL Executor 在 DuckDB 中执行查询。
9. Analysis 模块基于查询结果生成业务分析结论。
10. Chart 模块根据字段类型和数据形态推荐图表，并生成 ECharts 或 Plotly 配置。
11. History 模块保存问题、SQL、结果摘要、图表配置和耗时。
12. 前端展示 SQL、表格、图表和分析结论。

## API 设计

### `POST /api/ask`

请求：

```json
{
  "question": "2026 年 4 月各区域 XC60 销量排名如何？",
  "session_id": "demo-session",
  "top_k": 5
}
```

响应：

```json
{
  "question": "2026 年 4 月各区域 XC60 销量排名如何？",
  "intent": "sales_analysis",
  "sql": "SELECT region, SUM(units_sold) AS total_units FROM fact_sales ...",
  "columns": ["region", "total_units"],
  "rows": [
    {"region": "华东", "total_units": 328},
    {"region": "华南", "total_units": 261}
  ],
  "analysis": "2026 年 4 月 XC60 在华东区域销量最高，达到 328 台。",
  "chart": {
    "chart_type": "bar",
    "x_field": "region",
    "y_field": "total_units",
    "title": "2026 年 4 月各区域 XC60 销量排名"
  },
  "safety": {
    "is_safe": true,
    "risks": []
  }
}
```

## 模块职责

### Intent Service

识别问题类型和业务实体，包括车型、地区、时间范围、指标、渠道等。

### RAG Service

从数据字典和指标口径中检索相关上下文，降低 Text-to-SQL 的幻觉风险。

### Text-to-SQL Service

根据用户问题和检索上下文生成只读 SQL。该模块不直接执行 SQL。

### SQL Guard

负责 SQL 安全校验：

- 只允许 SELECT。
- 禁止 DDL / DML。
- 限制表白名单。
- 禁止访问敏感字段。
- 自动补充 LIMIT。

### SQL Executor

连接 DuckDB 或 SQLite，执行通过校验的 SQL，返回结构化数据。

### Analysis Service

基于查询结果生成业务摘要，输出核心结论、关键依据和行动建议。

### Chart Service

根据问题类型和结果字段推荐图表类型：

- 时间趋势：折线图。
- 分类对比：柱状图。
- 占比结构：饼图。
- 转化链路：漏斗图。
- 单指标结果：指标卡。

### History Service

记录用户问题、意图、SQL、分析结论、图表配置、执行耗时和错误信息，便于复盘和测试。

## 异常处理

| 场景 | 处理方式 |
| --- | --- |
| 意图无法识别 | 返回澄清问题，提示用户补充时间、车型或指标 |
| RAG 无相关文档 | 使用基础数据字典，并在响应中标记上下文不足 |
| SQL 校验失败 | 拦截执行，返回风险原因 |
| 查询结果为空 | 返回空结果说明，并建议扩大时间范围或调整条件 |
| LLM 响应格式错误 | 进行 JSON 修复或重试一次 |
| 数据库执行失败 | 返回 SQL 错误摘要，不暴露内部堆栈 |

## 最小可行版本边界

2-3 周内建议先完成：

- 5 张模拟数据表。
- 10-20 个标准业务问题。
- 一个 `/api/ask` 主接口。
- RAG 检索数据字典和指标口径。
- Text-to-SQL 生成与 SQL 安全校验。
- Streamlit 页面展示结果。
- pytest 覆盖核心安全与接口逻辑。

暂不建议做：

- 多用户权限系统。
- 企业级数据血缘。
- 复杂 Agent 多轮规划。
- 真实 BI 平台集成。
- 大规模分布式数据处理。
