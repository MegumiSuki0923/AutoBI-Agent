# 简历写法与面试讲解

## 简历项目经历

### AutoBI Agent：面向汽车营销数据平台的智能问数与分析助手

**技术栈：** Python、FastAPI、Pydantic、DuckDB、Pandas、LLM API、RAG、Text-to-SQL、sqlglot、Streamlit、ECharts/Plotly、pytest、Git

**项目描述：**  
面向汽车营销与销售数据场景，设计并实现企业级自然语言问数智能体原型。系统支持用户通过自然语言查询车型销量、线索转化、试驾预约、渠道投放等业务数据，并自动完成指标检索、SQL 生成、安全校验、数据查询、结果分析与图表展示，模拟企业数据平台中的 AI 问数与分析流程。

**项目亮点：**

- **接口设计：** 基于 FastAPI + Pydantic 设计问数接口，统一封装用户问题、SQL、查询结果、分析摘要和图表配置。
- **数据建模：** 构建汽车营销销售漏斗模拟数据集，覆盖投放、线索、试驾、成交等业务链路，沉淀数据字典和指标口径。
- **智能问数：** 实现自然语言问题理解与 Text-to-SQL 生成，使业务问题可自动转换为结构化数据查询。
- **RAG 检索：** 将表结构、字段说明、指标公式和示例 SQL 向量化检索，提升 SQL 生成的业务一致性。
- **结果分析：** 基于查询结果生成业务结论、关键数据依据和行动建议，提升问数结果的可解释性。
- **可视化：** 根据数据形态自动推荐折线图、柱状图、饼图或漏斗图，并生成前端可渲染图表配置。
- **测试验证：** 使用 pytest 构建问数测试集，覆盖典型业务问题、SQL 安全、空结果、多表 Join 和边界问题。
- **工程沉淀：** 沉淀 Prompt 模板、接口文档、指标口径文档和 AI Coding 开发记录，形成可复用工程资产。

## 面试可讲点

### 1. 为什么做 AutoBI Agent？

回答方向：

企业数据平台中，业务人员经常需要依赖数据分析师写 SQL 才能完成问数。AutoBI Agent 的目标是降低数据使用门槛，让业务人员用自然语言完成销售、线索、试驾和投放数据分析。

### 2. 为什么选择汽车营销销售场景？

回答方向：

这个场景与汽车企业业务贴近，并且天然包含投放、线索、试驾、成交等完整漏斗，既能体现业务理解，也能展示数据建模、指标口径和可视化分析能力。

### 3. 为什么不用真实企业数据？

回答方向：

真实企业数据涉及隐私和合规，因此项目使用模拟数据复现业务结构。重点不是数据本身，而是展示企业数据平台中从数据建模、问数接口、SQL 安全到文档沉淀的完整工程链路。

### 4. Text-to-SQL 怎么保证准确？

回答方向：

不是直接把问题丢给大模型，而是先通过 RAG 检索相关表结构、字段说明、指标口径和示例 SQL，再让模型基于受控上下文生成 SQL。这样可以减少模型编造字段和误用指标的问题。

### 5. SQL 安全怎么做？

回答方向：

使用 sqlglot 解析 SQL，只允许 SELECT，禁止 DROP、DELETE、UPDATE、INSERT、ALTER 等操作。同时限制表白名单，禁止访问敏感字段，并统一加 LIMIT，避免大结果集查询。

### 6. RAG 在项目里检索什么？

回答方向：

主要检索数据字典、指标口径、表关系、业务术语和示例 SQL。例如用户问“线索转化率”，系统会检索到这个指标的计算公式和相关字段，再生成 SQL。

### 7. 项目如何体现 AI Coding？

回答方向：

项目开发过程中使用 AI 辅助完成接口草稿、Prompt 初稿、测试用例和文档生成，但核心逻辑会通过人工审查和 pytest 验证。重点是把 AI 作为开发效率工具，而不是无校验地接受 AI 输出。

### 8. 如何评估问数智能体效果？

回答方向：

设计标准问题集，评估意图识别准确率、SQL 可执行率、SQL 安全拦截率、指标口径命中率、图表推荐准确率和分析摘要可信度。对于危险问题和模糊问题，也会设置单独测试。

### 9. 如果要上线到企业环境，你会怎么扩展？

回答方向：

需要接入真实 EDH 或数据仓库，增加统一权限系统、指标平台、日志监控、Prompt 版本管理和人工反馈闭环。同时需要把 SQL 执行放在受控服务中，并对高风险查询做审计。

### 10. 这个项目最大的工程难点是什么？

回答方向：

难点不是调用 LLM，而是让 LLM 生成的结果可控、可验证、可追溯。因此我把系统拆成 RAG 检索、SQL 生成、SQL 校验、SQL 执行、结果总结和测试集几个模块，保证每一层都可以单独验证。

## 一分钟项目介绍

我做了一个 AutoBI Agent，面向汽车营销销售数据平台，模拟企业内部的自然语言问数场景。用户可以直接问“4 月各区域 XC60 销量排名如何”或者“抖音投放 CPL 是否高于小红书”，系统会先检索数据字典和指标口径，再生成 SQL，并通过 SQL Guard 做只读和表白名单校验，最后执行查询、生成业务结论和图表配置。

这个项目的重点不是单纯调用大模型，而是把问数流程工程化：包括 FastAPI 接口、Pydantic 响应模型、DuckDB 数据集、RAG 上下文检索、Prompt 模板沉淀、pytest 测试集和技术文档。它比较贴近企业数据平台中的 AI 应用落地，也符合 AI Coding 实习中对开发效率、Prompt 规范、测试验证和文档沉淀的要求。
